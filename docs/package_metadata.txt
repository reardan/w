Package metadata design

Goal:
	Describe packages and modules well enough for tools to validate a W
	project, explain dependencies, and keep builds reproducible without changing
	the compiler's current import lookup or any floating-point, codegen, or type
	behavior.

Non-goals:
	Do not add compiler import search paths in this design.
	Do not change dotted import resolution: import a.b still maps to a/b.w from
	the compiler's active source root, with the existing __arch__ substitution.
	Do not add dependency fetching, floating-point metadata, code generation
	metadata, or type-system controls.

Metadata file:
	Each package root may contain package.wmeta.  The format is deliberately
	line-oriented so it can be parsed by W tooling later without needing a
	general-purpose data format parser.

	Blank lines and # comments are ignored.  Top-level fields use:

	package <dotted-name>
	version <major>.<minor>.<patch>
	w_language <constraint>
	root <relative-path>

	Sections end at the next section header:

	modules:
		<dotted.module>
		<dotted.module>

	dependencies:
		<package-name> <constraint>
		<package-name> <constraint> path <relative-path>

	Example:

	package w
	version 0.1.0
	w_language >=0.1.0
	root .

	modules:
		w
		compiler.compiler
		lib.lib
		structures.array_list

	dependencies:
		example.net >=1.2.0 path vendor/example.net

Field meaning:
	package is the canonical package name.  Use lower-case dotted names for
	published packages, matching the spelling used by dependency entries.

	version is a SemVer-style project version.  Pre-release and build metadata
	can be reserved for a later package registry; initial tooling should accept
	only three decimal components to keep comparison simple.

	w_language is the minimum or bounded W language version the package expects.
	It describes source compatibility only.  It must not enable language
	features, change warnings, or switch codegen paths.

	root is the package's source root relative to package.wmeta.  The first
	stage should require ".".  Allowing another relative root later gives tools a
	place to represent repos that keep W sources under src/ without changing the
	compiler.

	modules is a manifest of public or buildable modules owned by the package.
	Each entry maps to the current filesystem rule under root.  For example,
	compiler.compiler means compiler/compiler.w.

	dependencies lists the package graph expected by build tooling.  The optional
	path field points at an already-present checkout or vendored copy.  It is not
	a compiler search path; it is an input for future tooling that can validate
	or assemble an explicit build tree before invoking the compiler.

Dependency and version semantics:
	Package identity is the package field, not the directory name.

	Versions compare numerically by major, minor, then patch.  The initial
	constraints should be:
		=1.2.3 exact version
		>=1.2.3 minimum version
		^1.2.3 compatible major range, equivalent to >=1.2.3 and <2.0.0
		^0.2.3 compatible minor range, equivalent to >=0.2.3 and <0.3.0

	A dependency is satisfied when the referenced package.wmeta package name
	matches and its version satisfies the constraint.  If two dependencies select
	the same package name with incompatible constraints, tooling should fail
	before compilation.  The compiler should not try to choose between versions.

	Transitive dependencies are resolved by tooling from metadata files, then
	recorded in a future lock file.  The lock file should contain package name,
	version, source location, and content hash.  It is intentionally separate
	from package.wmeta so project intent and machine resolution remain distinct.

Import path implications:
	Current W source imports remain unchanged.  Metadata can say which modules a
	package owns and where dependency packages are located, but source files
	should still import dotted modules exactly as they do today.

	The first implementation should validate only that declared modules exist
	under root using the current mapping.  It should not teach the compiler to
	search dependency directories.  If a later build tool wants external package
	imports, it can create an explicit temporary or vendored tree whose paths
	already match the current import mapping, then run the compiler normally.

	Reserved segments keep their current meaning.  __arch__ is still resolved by
	the compiler target, and metadata must not redefine it.  __word_size__ stays a
	compile-time constant and is not a metadata variable.

Makefile and build integration:
	The bootstrap path should stay unchanged:
		make build
		make verify
		make tests

	Initial metadata support should be additive:
		make metadata_check

	metadata_check can validate package.wmeta syntax, version constraints,
	module existence, duplicate module declarations, and dependency metadata
	for any path dependencies present in the tree.  It should not be required by
	build or verify until the checker is self-hosted enough to avoid making the
	seed compiler bootstrap depend on new tooling.

	Once stable, make tests can depend on metadata_check.  make build should
	continue compiling w.w directly so bootstrapping remains understandable and
	package metadata cannot hide a compiler dependency cycle.

	Distribution targets can later package sources plus package.wmeta, while
	lock-file generation can remain a separate target such as make package_lock.

Staged implementation:
	1. Land this design and a sample package.wmeta fixture in tests/metadata/ or
		docs/examples/ if examples become useful.
	2. Add a standalone metadata parser/checker that validates one package file
		and path dependencies, with no compiler integration.
	3. Add package.wmeta for the root repo and any standard-library subpackages
		only after the checker can prove module paths exist.
	4. Add make metadata_check and include it in make tests after it is stable.
	5. Add optional lock-file generation for path dependencies and content
		hashes.
	6. Revisit import ergonomics only after metadata validation is useful on its
		own; any future import changes should be a separate compiler design.
