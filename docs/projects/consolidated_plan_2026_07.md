# Consolidated plan: manifest de-churn, REPL, and the VCS/build workstream

Status: plan (July 2026). Successor to
`docs/projects/issue_audit_2026_07.md`, whose waves 0–4 are complete
(18 PRs, #250–#274). Inputs: issues #251 (build system) and #252 (VCS),
the maintainer's REPL research (`docs/projects/repl_improvements.md`),
and the churn observed while executing the wave program. Execution model
is unchanged: waves of parallel subagent PRs, sequential merge after
green, Sonnet for well-specified/leaf work, Fable for compiler/debugger
internals and investigations.

## 1. What the wave program taught us about churn

Evidence from waves 0–4 (18 merged PRs):

- **`build.json`** (generated, committed) was regenerated by roughly 10
  of the 18 PRs. Three merges needed manual rebases whose only real
  content was `./wbuild manifest` (#260, #267, #274).
- **`wtest_map_test`'s frozen expected-selection lists** — exact-set
  `stdin` golden lists inside `build.base.json` — were the single worst
  conflict surface: at least six PRs edited them (#259, #260, #271,
  #272, #274 among them), every edit is order-sensitive against
  `bin/wtest`'s real output, and one union-merge went wrong on a
  substring (`file_test` inside `missing_file_test`) before the fixture
  itself caught it. #268 additionally found and fixed a selection bug
  (`wtest_doc_only` swallowing `tests/asm/*.txt`), i.e. the frozen lists
  were guarding a mechanism that itself had a hole.
- **`build.base.json`** (1,784 lines, 191 `stdin` entries at head) grew
  hand-written steps in 8 PRs (repl/debug session scripts, trap
  `expect_stderr`, gated openssl target, fixture runners).

These map directly onto #251 Direction 1 ("compiler as the single source
of truth"). Conclusion: **do Direction 1 first**, before the VCS waves —
every later wave, including #252's, merges more cheaply afterwards.

        \                            /
         → merge all sequentially ←
                   ↓
Wave A3∥A4 (deps keys, platform axis)   Wave R3 (REPL split, alone)
                   ↓                            ↓
Wave V1a∥V1b∥V1c (VCS foundations)      Wave R4∥R5 (#114, wdbg core)
                   ↓
        D3-1 → Wave V2a∥V2b → V2c → D3-2 → 4a → (V3, V4, 4c, 4b …)

Threads A/R are file-disjoint (tools/manifests vs repl/debugger) and run
in parallel; Thread C waits for Wave A because its every PR adds
tests/targets and should land into the de-churned manifest world. The
`build.json`-conflict rule (regenerate, never hand-merge) stays in force
until A1/A2 land, after which it should rarely trigger.

## 6. Execution-protocol updates (lessons from waves 0–4)

- Keep: sequential merge after green; rebase + regenerate manifest on
  conflict; warning-fixture changes bundled one-PR-per-wave; seed-graph
  PRs merge last and alone; worktree isolation per agent.
- New: agents must run long test suites in the **foreground** (with
  generous timeouts), not as fire-and-forget background jobs — the
  "stopped waiting for its own suite" stall hit four agents and cost
  wall-clock time each time.
- New: any fixture that freezes ordered output must state, in a comment
  next to it, the exact command that regenerates it (A1 generalizes
  this).
- Carry-over environment facts: qemu-user-static and libc6:i386 are
  installed in this container, so `verify_arm64` and the 32-bit dynamic
  tests run here; darwin validation still needs the Mac.
